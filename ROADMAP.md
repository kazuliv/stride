# Stride ロードマップ

## 思想・コンセプト
Stride = 自分のリズムをコントロールし、スローダウンを意識的に進めるタスク管理ツール
The Tuningと同じ思想。やみくもに生産性を上げるのではなく、自分の歩幅(stride)で進む。

---

## Phase 1: 自前カレンダー（iframe→自前UI置換）

### 目的
- 自分の時間の使い方を「見える化」し、リズムを整える
- 稼働ログをカレンダー上で直感的に記録・確認
- GCalとの双方向同期を維持しつつ、Stride独自の体験を提供

### 実装内容

#### 1-1. カレンダー計画ビューの上部ペインを自前カレンダーに置換
- 既存の`buildHomeWeekGrid`をベースに拡張
- 日/週/月表示の切り替え
- 時間軸グリッド（8:00-24:00）
- 表示要素:
  - Strideタスク（アクション日/時間）→ 企業カラーで表示
  - 稼働ログ（actionLogs）→ 半透明のブロックで視覚化
  - GCalイベント（state.gcalEvents）→ 別色で表示
  - 合計稼働時間のサマリー表示

#### 1-2. カレンダー上での操作
- 空き時間クリック → 稼働ログ追加 or タスクのアクション日設定
- ドラッグで時間変更（Stride→GCal自動反映）
- イベントクリック → タスク詳細表示

#### 1-3. GCal双方向同期（既存基盤を活用）
- Stride → GCal: タスク/ログ作成・変更時にAPI呼び出し（実装済み）
- GCal → Stride: fetchGcalEvents + syncGcalChangesToStride（実装済み）
- 制限: GCal側変更のStride反映はポーリング（ページ遷移/リロード時）
- 将来: サーバー構築後にWebhookでリアルタイム化可能

#### 1-4. リズム可視化（Stride思想の体現）
- 日/週の稼働時間サマリー
- 「今日の余白」表示（予定のない時間の可視化）
- 稼働ペースのトレンド（週ごとの稼働時間推移）
- 休憩・バッファ時間の意識的な確保を促すUI

### 技術メモ
- ホーム画面のGCal embedも自前カレンダーに統一するか検討
- iframe embedは色の制御が難しい問題があった（既知の制限）
- buildHomeWeekGridに既にタスク/GCalイベント/時間指定の描画ロジックあり

## Phase 2: チーム機能（バックエンド構築）

### 目的
- 少数チームでのタスク共有
- GTD Inboxの「委任」をチームメンバーへの実際の委任に進化
- チーム全体のリズムを整える

### 技術構成（推奨）
- Firebase（Firestore + Auth + Hosting）が最速
  - リアルタイム同期が標準装備
  - 認証が容易
- 代替: Next.js + Supabase（自由度重視）

### 実装内容

#### 2-1. バックエンド移行
- localStorage → Firestore（or Supabase）
- データ移行ツール（既存localStorageデータのインポート）
- オフライン対応（Firestoreのオフラインキャッシュ）

#### 2-2. 認証・チーム管理
- Google OAuth認証（GCal連携と統合可能）
- チーム作成・メンバー招待
- 権限管理（オーナー/メンバー/閲覧者）

#### 2-3. タスク共有
- チーム共有の企業・プロジェクト
- タスクのアサイン（担当者設定）
- 委任 = チームメンバーへの実タスク移譲
  - 委任されたタスクは相手のInboxに表示
  - 確認日の自動リマインド

#### 2-4. チームのリズム可視化（Stride思想）
- チームメンバーの稼働状況（負荷の偏り検知）
- 「このメンバーに余白はあるか」の可視化
- チーム全体のペース管理ダッシュボード

#### 2-5. リアルタイム同期
- Firestoreのリアルタイムリスナー
- GCal Webhook通知（サーバー経由）
- タスク更新の即時反映

### 段階的移行案
```
Step 1: Firebase Hosting + Auth のみ（現行コードをそのままホスティング）
Step 2: Firestore移行（localStorageからDB化）
Step 3: チーム機能追加
Step 4: GCal Webhook対応（リアルタイム双方向同期）
```

## グローバルショートカットについて
- Webアプリのままではブラウザ非アクティブ時に不可
- PWA化である程度対応可能（Dock追加、オフライン）
- 必要なら軽量Electronランチャー（Quick Inbox専用）+ Webアプリ本体の構成も可
- チーム機能との兼ね合いでWebアプリ維持が望ましい
